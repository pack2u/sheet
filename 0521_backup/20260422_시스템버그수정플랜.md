# 시스템 버그 및 성능 개선 수정 플랜

이전에 보고된 `독립배포시트 시스템 점검 보고서`를 기반으로, 실제 코드에 반영할 수정 계획을 정리했습니다.

## User Review Required

> [!IMPORTANT]
> 아래 수정 사항들은 시스템의 크래시를 방지하고 성능을 크게 향상시킵니다. 특히 `pushOrdersToOldSystem.gs`의 버그는 통합 전송 기능 마비를 일으키므로 즉시 수정이 필요합니다. 코드를 수정해도 괜찮을지 승인해 주시면 바로 작업을 진행하겠습니다.

## Proposed Changes

### 1. pushOrdersToOldSystem.gs

**버그 수정: 배열 선언 누락으로 인한 크래시 방지**
- `pushHubOrdersToOldSystem` 함수 내에서 발주 데이터를 담을 `row` 배열의 선언이 누락된 것을 수정합니다.
- 루프 내부, 데이터 푸시 시작 전 위치에 `var row = [];` 를 추가하여 ReferenceError 를 해결합니다.

#### [MODIFY] pushOrdersToOldSystem.gs
- 58번 라인 근처:
```javascript
       var pInfo = infoMap[itemCodeNum] || { status: "", loc: "" };
       
       var row = []; // 추가할 코드
       
       // A. 순번 (빈칸 - 구글시트 수식 또는 수동입력으로 처리)
       row.push(""); 
```

### 2. systemAnalyzer.gs

**성능 개선: 무한 루프로 인한 타임아웃 방지**
- `runSystemAnalyzer` 함수 내 폴더 전수 스캔 로직에서 파일이 너무 많을 경우 스크립트 실행 시간 초과(Time Out)가 발생할 수 있습니다.
- 스캔할 최대 파일 개수에 제한(예: 50개)을 두어 타임아웃을 방지합니다.

#### [MODIFY] systemAnalyzer.gs
- 91번 라인 근처 `while(files.hasNext())` 루프 수정:
```javascript
    var fileCount = 0;
    var maxLimit = 50; // 최대 검색 파일 수 제한 (추가)
    while(files.hasNext()) {
      if (fileCount >= maxLimit) {
        report.push(["  (⚠️ 파일이 너무 많아 처음 " + maxLimit + "개까지만 스캔했습니다.)"]);
        break; // 루프 탈출
      }
      var file = files.next();
      if (file.getId() === ss.getId()) continue; // 현재 파일 제외
      report.push(["  - 연결된 뷰어 파일: " + file.getName() + " (ID: " + file.getId() + ")"]);
      fileCount++;
    }
```

### 3. ecount.gs

**최적화: 불필요한 flush 렌더링 부하 최소화**
- 백그라운드 스케줄링 시 화면 렌더링 업데이트 함수인 `SpreadsheetApp.flush()`가 필요 이상으로 동작하여 성능 저하를 일으킬 수 있습니다.
- 필수적인 위치 이외의 `flush()` 호출을 최소화하여 성능을 최적화합니다.

#### [MODIFY] ecount.gs
- `ecountStep2`와 `ecountStep3` 내부의 `SpreadsheetApp.flush()` 호출 점검 및 최소화 로직 적용.
