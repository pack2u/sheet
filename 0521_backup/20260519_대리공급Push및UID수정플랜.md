# 20260519 대리공급 Push 및 UID 수정 플랜

## 1. 발생 원인 분석
- **P열 고유아이디(UID) 빈칸 문제**: 임시기록시트(대리발송 소스)로 데이터를 불러올 때 P열이 비어 있으면 스크립트가 자동으로 랜덤한 고유아이디(UUID)를 생성하여 부여하고 있었습니다.
- **자동 Push 중복 발주 발생**: 현재 '대리발주 자동 Push' 기능이 켜져 있어 5분마다 스크립트가 자동으로 실행되고 있었습니다. 그런데 어떤 이유로 원본 소스의 P열에 생성된 고유아이디가 정상적으로 저장(기록)되지 않거나 지워지면, 다음 5분 뒤 실행 시 **새로운 랜덤 고유아이디**가 또다시 생성됩니다. 
- 이로 인해 시스템은 '기존에 Push된 주문'이 아닌 '완전히 새로운 주문'으로 인식하게 되어 전용양식으로 똑같은 발주를 계속 중복해서 밀어넣는 현상(무한 증식)이 발생했습니다.

## 2. 해결 방안 (반영 완료)
1. **고정형(Deterministic) 고유아이디 자동 생성 알고리즘 도입**
   - 기존의 무작위 랜덤 문자열(UUID) 방식 대신, 발주 데이터 자체(일자, 품목코드, 수량, 전화번호, 주소)를 조합하여 암호화(MD5 해시)하는 방식을 적용했습니다.
   - 이제 P열이 비어 있더라도 해당 주문의 내용이 똑같으면 **항상 똑같은 고유아이디(예: A-1F2B3C4D)** 가 생성됩니다.
   - 따라서 원본 시트에 UID 저장이 실패하더라도, 다음 자동 실행 때 똑같은 UID가 만들어지므로 시스템이 "아, 이거 이미 Push한 건이네!" 하고 정상적으로 스킵(Skip)하게 되어 **중복 발주가 원천 차단**됩니다.

## 3. 사용자 조치 사항 (안내)
- 자동 Push로 인해 발주가 계속 들어오는 현상은 위 코드 수정으로 즉시 해결(중복 차단)됩니다.
- 하지만 만약 **5분마다 자동으로 구동되는 것 자체를 멈추고 수동으로만 제어하고 싶으시다면**, 스프레드시트 상단 메뉴에서 **[대리발주 자동 Push 끄기]** 버튼을 클릭해주시면 자동 구동이 즉시 중지됩니다.
