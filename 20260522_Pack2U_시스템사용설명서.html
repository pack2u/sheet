시스템 아키텍처 개요

Pack2U 구글 스프레드시트와 협력업체 개별 발주서 간의 데이터를 전송, 수집, 마감하는 연동 시스템의 명세서 및 사용 설명서입니다.

시스템 구조와 데이터 흐름

본 시스템은 중앙의 상품정보 스프레드시트를 허브로 작동하며, 다수의 협력업체 개별 스프레드시트와 실시간/주기적 API 통신을 통해 다음 기능을 수행합니다.

1

발주 데이터 1차 집합 (대리공급업체 발주 탭)

수동 또는 외부 ERP 등에서 대리공급 대상 주문 정보가 대리공급업체 발주 시트에 입력됩니다.

2

업체별 자동 분류 및 Push

이카운트코드 분석 및 품목 매핑 후, 각 협력업체 고유 파일의 전용양식 탭에 자동으로 나뉘어 전송(Push)됩니다. 동시에 중앙 시트의 대리공급_임시기록 시트에도 상태가 실시간 백업됩니다.

3

업체 송장 입력 및 역수집

협력업체가 전송받은 파일에 직접 송장번호를 채우면, 연동 트리거가 이를 감지하여 중앙의 협력업체_발주허브 및 대리공급_임시기록의 송장번호를 자동 업데이트합니다.

4

ERP 판매현황 반영 및 월별 마감

송장이 입력된 발주 데이터는 판매현황 시트로 모여 이카운트 ERP로 최종 업로드되고, 마감 처리를 거치며 원본 시트에서 아카이브 탭으로 이동되어 다음 날 발주를 위한 공백 상태를 확보합니다.

주요 관리 시트 정의
시트/탭 이름 주요 역할 및 관리 정보 비고 및 특이사항
대리공급업체 발주 모든 대리공급 및 위탁 발주서의 1차 입력 소스 시트 D열의 이카운트코드가 핵심 식별값
협력업체_발주허브 송장 수집 및 판매현황 연동의 중앙 DB 역할 P열: 이카운트 업 완료 여부, Q열: 진행 상태
대리공급_임시기록 사방넷 주문번호 기준 당일 대리공급 발송 흐름 모니터링 시트 X열: 송장번호, Y열: 진행상태 ("송장수집", "발주완료")
대리발송 별칭 테이블 원본 이카운트 품목코드를 협력업체 전용 코드로 매핑해주는 별칭 매칭표 이 매핑 데이터가 없으면 전용 규격 변환이 불가
1. 대리공급업체 발주 Push
📂 관련 스크립트 파일
_partnerExclusivePush.gs → partnerPushOrdersToExclusiveForms()

사용자 또는 자동 트리거(5분 간격)에 의해 대리공급업체 발주 탭의 새로운 건이 처리됩니다.

  • 업체 식별 기준: D열(이카운트코드)의 앞 2자리(예: NK, HR, BW)를 추출해 식별된 협력업체 스프레드시트의 파일 ID를 연동 맵에서 조회합니다.
  • 품목 코드 변환 (별칭 적용): 대리발송 별칭 테이블에서 원본 품목코드를 검색하여 해당 업체에서 실제 접수하는 업체용 품목코드(vendorSku) 및 품목명(vendorName)으로 치환합니다.
  • 중복 Push 방지: 사방넷주문번호(UID) + 치환된 품목코드를 조합한 복합 고유키(UID|품목코드)를 생성합니다. 협력업체 전용 시트의 데이터를 읽어와 이미 전송된 고유키와 비교한 뒤, 기존에 내보낸 적 없는 새로운 행만 전송합니다.
  • 임시기록 생성: Push된 데이터는 대리공급_임시기록 시트 하단에 즉시 덧붙여 기록됩니다. (P열에 주문번호(UID), W열에 업체 코드, Y열에 "발주완료" 표시 기입)
2. 송장번호 역수집
📂 관련 스크립트 파일
_partnerOrders.gs → partnerFetchInvoices()

협력업체가 전용 시트의 A열(송장번호)에 직접 작성한 발송 정보를 수집합니다.

  • 수집 대상: 모든 협력업체의 전용 발주 스프레드시트 전용양식 탭을 순회 탐색합니다.
  • 매칭 및 동기화: 전용 시트 AX열(50번째 열)에 저장되어 있던 고유ID(UID)를 키로 활용해 중앙 메인 시트에서 해당 주문을 찾습니다.
  • 기입 동작:
    - 메인 시트 협력업체_발주허브의 N열(송장번호)을 채워 넣고 상태(O열)를 "배송중" 또는 "발송완료"로 갱신합니다.
    - 메인 시트 대리공급_임시기록의 X열(송장번호)에 값을 넣고 Y열(진행상태)을 송장수집으로 전환합니다.
판매현황 갱신 및 이중 갱신 방지 (ERP 업로드 준비)
📂 관련 스크립트 파일
_partnerOrders.gs → partnerRebuildSalesUploadSheetCore_()

이카운트 ERP 판매 입력용 원본 시트를 갱신하는 기능의 작동 규격입니다.

A

허브 데이터 필터링

협력업체_발주허브 탭의 전체 행을 읽어옵니다. 이때 **P열(이카운트 업 완료 여부)에 "이카운트 업 완료"라는 문구가 이미 적힌 행은 수집 대상에서 제외(스킵)**하여 이중 전송이나 중복 매출 기입을 방지합니다.

B

갱신 대상 데이터 빌드

필터링을 통과한 유효 발주 데이터를 이카운트 ERP 판매업로드 양식 형식에 맞추어 `판매현황_갱신` 시트 데이터로 가공 및 재정렬하여 써넣습니다.

C

상태 플래그 업데이트

갱신 처리가 정상적으로 완수되면, 방금 수집하여 처리한 협력업체_발주허브 행들의 P열에 즉시 이카운트 업 완료 텍스트를 기입하여 락(Lock)을 겁니다. (최초 실행 시 P열 1행에 '이카운트 업 완료 여부' 헤더 자동 생성)

일괄 마감 업데이트 처리 흐름
📂 관련 스크립트 파일
_partnerWebApp.gs → partnerDailyArchiveAll()

일괄 마감 업데이트를 실행하면 다음 4가지 마감 스크립트가 순서대로 연쇄 작동하여 데이터 보관 및 원본 초기화를 수행합니다.

순서 기능 단위 대상 데이터 조건 마감 시 보관 경로 원본 시트 처리
1 월별 정산 이동
(Monthly Settle)		 오늘 이전 날짜 & **송장번호 입력 완료** 건 협력업체 파일 내
(YYYY년 M월) 발주 마감 탭		 원본 발주 및 송장조회 탭에서 해당 데이터만 삭제
(미입력 건은 계속 잔류)
2 전용 마감 이동
(Exclusive Archive)		 오늘 이전 날짜 & (**송장 있음** 또는 **발송완료** 건) 협력업체 파일 내
(YYYY년 M월) 전용발주 마감 탭		 원본 전용양식 탭에서 해당 행 삭제
(미완료 건 잔류 및 소스 시트 협력Push ID 공백 리셋)
3 통합 허브 아카이브
(Hub Archive)		 오늘 이전 날짜 & (**송장 있음** 또는 **취소/품절/단종/발송완료** 건) 지정 구글 드라이브 폴더의
[Pack2U 통합발주 아카이브] YYYY-MM 파일		 메인 시트의 협력업체_발주허브 원본 탭에서 아카이브가 완료된 행 삭제
4 임시 기록 청소
(Temp Archive)		 **송장번호(X열)가 정상 기입**된 완료 행 (백업 없이 단순 청소) 메인 시트의 대리공급_임시기록 시트에서 송장이 매칭 완료된 행들만 골라 지움
(송장이 아직 수집 안 된 건은 계속 유지)
향후 시스템 고도화를 위한 편의 기능 제안

Pack2U 발주서-송장 흐름의 안정성과 운영 편의성을 대폭 향상할 수 있는 추가 고도화 기능 제안 목록입니다.

1

송장 미기입 장기 방치 주문 리마인더 (알림 시스템)

상세 구성: 발주 후 3일(영업일 기준) 이상 경과하였음에도 송장번호(운송장)가 등록되지 않은 발주 미발송 건이 있을 경우, 스프레드시트 관리자 화면 상단에 경고 배너를 띄우거나 담당자에게 슬랙/카카오톡으로 일일 미발송 알림 리포트를 자동 전송하는 기능입니다.

장점: 협력업체의 누락 발주 및 출고 지연 사고를 선제적으로 모니터링하여 고객 불만을 대폭 감소시킵니다.

2

이카운트 ERP 판매현황 갱신 자동 자동화 스케줄러

상세 구성: 관리자가 매번 시트에서 수동으로 '판매현황 갱신' 버튼을 누르지 않아도, 설정된 마감 시간(예: 오후 6시)에 맞추어 하루 동안 수집된 완료 발주 데이터를 바탕으로 이카운트 업로드 양식을 자동 생성하고, 백업본을 드라이브에 저장해 두는 타임 트리거 스케줄링 기능입니다.

장점: 매일 수동으로 작업해야 하는 단순 반복적인 마감 단계를 1단계 줄여 운영 리소스를 단축합니다.

3

발주 오류/매칭 실패 시각화 통합 대시보드

상세 구성: 발주 Push 동작 시 '매핑 없음(이카운트코드 불일치)', '파일 없음(경로 오류)' 등으로 스킵되는 에러 건들이 발생했을 때, 스크립트 실행 로그(Logger)를 직접 열어보지 않고도 별도의 발주 오류 대시보드 시트 탭에 에러 행과 구체적인 실패 사유가 실시간 누적되어 기록되도록 구현하는 것입니다. 사용자는 이 화면에서 잘못된 정보를 즉시 확인하고 바로 수정할 수 있습니다.

장점: 발주 처리 누락을 조기에 감지하고 오류 복구 시간을 획기적으로 줄여 운영 안정성을 제공합니다.
