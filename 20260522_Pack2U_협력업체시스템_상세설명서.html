1. System Architecture Specification

1. 시스템 아키텍처 및 연동 흐름

중앙 데이터 허브와 각 협력업체 배포 시트 간의 데이터 통신, API 통합 및 트랜잭션 동기화 사양 문서입니다.

시스템 관계도 및 데이터 바인딩

본 시스템은 중앙 제어용 Pack2U 상품정보 스프레드시트와 협력업체가 운영하는 독립적인 배포 스프레드시트들을 유기적으로 결합한 다중 인스턴스 구조입니다. 보안 취약성을 극복하고 가볍고 빠른 응답성을 보장하기 위해 설계되었습니다.

🔒 보안 격리 설계 (Anti-Leak Policy)
과거에는 모든 업체 매핑 정보가 담긴 시트를 IMPORTRANGE로 불러왔으나, 타 업체 거래처명/파일 ID가 유출될 위험이 존재했습니다.
현재는 로컬 파일 내 설정 탭(B5 셀)에서 자기 업체의 거래처명만 직접 읽어오는 방식으로 개선되어 정보 유출 경로를 원천 차단했습니다.
1
중앙 데이터 기지 (허브 파일 구축)

상품정보시트에서 createStaticHub()를 구동하여 중앙 허브를 생성하며, 모든 업체별 그룹 단가(VIP, A, B 등)와 소비자가격, 상태(품절/단종/재고까지만)를 1행부터 기록 및 통합 유지합니다.

2
독립 배포시트 동기화 (IMPORTRANGE 바인딩)

배포된 협력업체 시트의 뷰어는 중앙 단가표를 실시간 IMPORTRANGE로 로드합니다. 이때, 뷰어 3행에 숨겨진 ARRAYFORMULA들이 이카운트 코드를 기반으로 품목명, 단가, 재고를 자동 출력합니다.

3
실시간 발주 트랜잭션 수집 (Fetch & Push)

5분 주기의 시간 트리거와 수동 매뉴얼 제어를 통해 중앙에서 업체별 미처리 발주를 수집 후 각 업체 전용 파일에 분배 기입합니다. 이후 업체가 송장을 채우면 이를 역수집하여 ERP 연동을 마칩니다.

협력업체 배포용 파일 내부 탭 사양

협력업체에 생성/배포되는 시트 파일 내부에는 역할에 따라 격리된 4개의 기본 시트 탭이 활성화되어 작동합니다.

1. [업체명] 뷰어 (단가 및 상태 조회 탭)

중앙 허브의 단가표와 1:1 결합되는 탭입니다. 2행 헤더 영역과 3행 수식 영역으로 나뉘며 3행은 외부 편집 방지를 위해 숨김 및 시트 보호가 적용되어 있습니다.

열 위치 헤더 명칭 동작 제어 (3행 주입 수식 및 논리)
A열 상태 =ARRAYFORMULA(IF(C3:C="", "", IFNA(XLOOKUP(C3:C, IDs, Hub!A:A)), "-"))) (품절, 단종 표시)
B열 출고지 =ARRAYFORMULA(IF(C3:C="", "", IFNA(XLOOKUP(C3:C, IDs, Hub!B:B)), "-")))
C열 이카운트코드 수동 입력 열 사용자가 직접 입력/검색하는 필드
D열 품목명 이카운트코드를 인덱스로 허브 품목명 자동 바인딩
E열 재고 허브의 해당 품목 재고 수량 연동
F열 소비자가 허브의 소비자가 연동
G열 최종단가 일반형: K2에 입력된 그룹 열 인덱스를 동적으로 치환해 해당 그룹 단가 표출
소비자용: 소비자가(F열) * 할인율 계산 후 ROUNDUP(..., -2)로 100원 단위 올림 처리
H열 단가변동 최종단가(G) - 지난단가(I) 계산값 (변동 없을 시 "-" 출력)
I열 지난단가 K2 + 2열을 타겟으로 지난 시즌 단가 연동
J열 익월변동단가 K2 + 4열의 예고 단가 표출 (변동값 없을 시 "-" 출력)
2. 발주 및 송장조회 (표준 발주 입력 탭)

협력업체가 직접 발주를 넣고 송장번호를 피드백하는 표준 규격 시트입니다. **A열(거래처명), B열(주문일자), L열(정산금액), M열(고유ID), N열(상태)**은 뷰어 및 메타 데이터를 참조하여 자동으로 채워집니다.

⚡ Spill 수식 주입 사양 (_pt_injectOrderSpillFormulas)
• A열: =ARRAYFORMULA(IF(C2:C="", "", AA1)) (AA1 메타 셀의 업체명을 수집)
• L열: =ARRAYFORMULA(IF(C2:C="", "", IFERROR(E2:E * XLOOKUP(C2:C, 뷰어!C:C, 뷰어!G:G, 0), 0))) (수량 × 뷰어의 해당 단가 실시간 연산)
• N열: =ARRAYFORMULA(IF(C2:C="", "", IFERROR(XLOOKUP(C2:C, 뷰어!C:C, 뷰어!A:A), "-"))) (뷰어 상태 컬럼 참조)
소비자용 배포시트의 가상 단가 제어 (K2 NNN 기법)

소비자가 기반 할인 시트의 경우, 매번 수식을 개별 작성하지 않고 **K2 셀의 값을 활용해 할인율을 식별**하도록 고안되었습니다.

  • 소비자용 시트는 K2 셀에 할인율 * 111 형태로 저장됩니다. (예: 5% 할인인 경우 K2 = 555, 7.5%인 경우 K2 = 777)
  • 수식 복구 및 재생성 스크립트 구동 시, K2 값을 바탕으로 할인율을 계산(dcRate = Math.round(K2 / 111))하여 3행의 최종단가(G열)에 곱해질 수식을 실시간 갱신합니다.
1. 대리공급업체 발주 Push 상세 명세

이카운트 양식의 발주 원천 데이터를 분석하여 협력업체별 전용양식 탭으로 분기 주입하는 정밀 트랜잭션 과정입니다.

1
업체 파일 수신처 확인 (Prefix Mapping)

D열의 품목코드 맨 앞 2자리(예: NK, HR, BW)를 추출해 매핑 딕셔너리(_PEP_VENDOR_DIRECT_MAP_)와 드라이브 경로를 교차 탐색하여 수신할 타겟 파일 ID를 식별합니다.

2
중복 발주 필터링 (Composite Dedup Key)

사방넷주문번호(UID)와 품목 별칭 테이블에서 치환된 협력업체 품목코드를 결합하여 복합 고유키(UID|코드)를 조합합니다. 대상 시트의 49번째 열(AX열)에 이미 존재하는 동일 키가 있을 경우, 중복 접수로 판단하여 전송에서 제외합니다.

3
전화번호 유실 방지 및 교대 색상 서식화

수취인 연락처 열을 숫자로 저장 시 앞자리의 "0"이 탈락되는 현상을 완벽 차단하기 위해, 해당 범위에 미리 setNumberFormat("@")을 일괄 지정합니다. 또한, 발송 배치를 시각적으로 구분하도록 매 Push 건마다 짝수/홀수 행 배경색(#efefef / #ffffff)을 다르게 지정합니다.

4
임시 기록 탭 백업 및 진행상태 표시

Push가 정상 처리되면 대리공급_임시기록 시트의 마지막 행 아래에 기록을 덧붙입니다. P열에 고유ID(UID), W열에 prefix, Y열에 진행상태인 **"발주완료"** 상태를 즉시 기록해 둡니다.

2. 송장번호 역수집 (Fetch Invoices)

각 협력업체가 자신의 배포 파일에 입력한 송장번호를 중앙 시트로 긁어와 이카운트 ERP 전표 업로드가 가능한 완성 상태로 가공하는 과정입니다.

  • 작동 함수: partnerFetchInvoices()가 주기적으로 활성화됩니다.
  • 역수집 키 매칭: 각 업체 시트의 전용양식 AX열(50번째 열)에 저장된 원본 고유 ID(UID)를 바탕으로 데이터 쌍을 맞춥니다.
  • 동기화 로직: 송장번호가 채워지는 즉시 중앙 협력업체_발주허브의 N열에 기입하며, 동시에 백업 시트인 대리공급_임시기록의 X열에 해당 송장번호를 써넣고 Y열의 진행상태를 송장수집으로 전환시킵니다.

연동 장애 발생 유형별 진단 및 조치법

운영 환경에서 발생할 수 있는 주요 예외 상황과 복구 툴 사용 방법입니다.

1. 다른 사무실이나 기기에서 연동이 차단될 때 (권한 오류)

구글 스프레드시트 API 연동은 실행 주체의 구글 계정 토큰을 거치기 때문에, 접속 장소나 주체가 바뀔 시 OAuth 보안 경고로 중단될 수 있습니다.

🚨 해결 프로세스
1. 스프레드시트 메뉴 상단에서 [🛠️ AS / 진단] → [🔧 복구 도구] → [🔑 스크립트 권한 승인 (직원 최초 1회)]를 실행합니다.
2. 구글 보안 인증 안내 창이 뜨면 계정을 선택하고, [고급] → [Pack2U 스크립트로 이동]을 눌러 접근 권한을 명시적으로 허용해 줍니다.
2. 단가조회 탭의 수식이 깨져서 단가/품목명이 안 나올 때

업체 사용자가 실수로 수식 영역을 건드렸거나 스케줄 꼬임으로 스필(Spill) 영역이 파괴된 경우입니다.

  • 특정 한 업체만 깨진 경우: 복구 메뉴에서 단가조회 탭 복구 (업체 선택)를 구동하여 해당 업체의 파일 아이디를 골라 수식만 강제 재주입합니다.
  • 모든 업체의 수식이 일시에 나간 경우: 복구 메뉴에서 🚨 단가조회 3행 수식 긴급 복구 (전체)를 실행해 연동된 모든 협력업체의 뷰어 시트를 초기 스키마로 원격 강제 리셋시킵니다.
3. 발주 Push 구동 시 발생하는 주요 에러 코드 목록
에러명 (Error Logs) 발생 사유 담당 관리자 조치법
매핑 없음 (No Map) D열의 품목코드 prefix(2자리)가 설정 매핑에 존재하지 않음 '대리발송 별칭 테이블' 또는 연동 모듈에 해당 prefix 바인딩 등록
파일 없음 (No File) 업체 ID에 해당하는 구글 드라이브 파일이 삭제되었거나 경로가 변경됨 구글 드라이브 휴지통을 확인하거나, 신규 시트 생성 후 ID를 매핑 탭에 갱신
락 선점 에러 (Lock Busy) 이전 아카이브나 Push 스크립트가 아직 돌고 있어 충돌 차단됨 5분 뒤 다시 시도하거나, 동기화 락 강제 해제 메뉴 실행
