OT. 시스템 핵심 연동 개념

Pack2U 중앙 스프레드시트와 협력업체가 따로 사용하는 개별 배포 시트 간에 어떻게 정보를 주고받는지 이해합니다.

중앙 시트와 협력업체 시트의 구조

Pack2U의 발주 관리 시스템은 본사의 중앙 시트(본체)와 외부 협력업체들이 사용하는 협력업체 배포 시트들로 분리되어 있습니다. 이는 거래처 간의 다른 정보가 유출되지 않도록 철저히 격리하면서도 필요한 데이터만 수집하기 위함입니다.

1
중앙 허브 구축 (본사)

본사 시트의 💰 통합 허브 단가 관리 메뉴에서 허브를 구성하면 모든 협력업체의 가격 정보가 담긴 단가 허브가 생성됩니다.

2
협력업체 시트 배포 및 연동 (업체)

각 협력업체 배포 시트의 단가조회 탭은 본사 허브와 연동됩니다. 직원이 협력업체 시트의 설정 탭 B5셀에 업체 이름을 입력하면, 해당 업체의 단가 그룹 정보만 안전하게 차단/필터링되어 불러와집니다.

3
발주 및 송장의 순환 흐름

대리판매 업체의 발주 수집 ➔ 이카운트 업로드 ➔ 대리공급 업체로의 발주 Push ➔ 송장 회수 ➔ 마감 프로세스로 이어지는 순환 고리로 기능합니다.

대표적인 세부 참조 시트 관계
시트 명칭 주요 역할 주요 연동 필드
협력업체_발주허브 본사의 발주 수집 본부 (데이터베이스) 고유ID, 이카운트코드, 정산금액, 송장번호, 상태
대리공급_임시기록 대리공급 발송 흐름의 일일 백업 P열: 사방넷 주문번호(UID), X열: 송장번호, Y열: 진행상태
발주 및 송장조회 각 협력업체 시트 내에 들어있는 발주서 입력 탭 A~K열: 수동 입력 및 단가조회 참조, L~N열: 수식 자동완성
대리발송 별칭 테이블 본사 이카운트코드 ➔ 협력업체용 상품코드 매핑 테이블 본사코드, 협력업체코드, 협력업체품목명
[자가 체크] 협력업체 시트가 본사에서 구축한 단가 데이터를 보안 문제 없이 불러오는 연동 기준은 무엇인가요?
제 1강

1강. 대리판매 발주수집

대리판매 협력업체들이 각자의 시트에 등록한 신규 주문들을 본사의 중앙 발주허브로 가져오는 단계입니다.

참조 시트
업체 시트: 발주 및 송장조회 탭 본사 시트: 협력업체_발주허브 탭
실행 조건
오전 발주 마감 시간 이후
※ 5분 간격 자동 수집 트리거도 작동 중
수행 후 결과
허브에 새 주문 누적
업체 시트 상태 ➔ "접수완료" 기입
동작 메커니즘

메뉴에서 1️⃣ 대리판매 발주수집을 클릭하면 시스템이 자동으로 다음과 같이 구동됩니다.

1
협력업체 배포 시트 자동 조회

구글 드라이브 지정 폴더 내에 있는 모든 '[협력업체] ...' 스프레드시트들을 하나씩 순회합니다.

2
미수집 발주 추출

각 협력업체 시트의 발주 및 송장조회 탭에서 N열(상태)이 '접수완료'가 아니고, C열(이카운트코드)이 빈칸이 아닌 행들을 신규 발주 데이터로 간주합니다.

3
중복체크 및 허브 적재

이미 본사의 협력업체_발주허브에 수집된 M열(고유ID)이 존재하지 않는 경우에만 본사 허브 시트의 최하단에 행을 덧붙여 수집합니다.

4
역기록(접수완료)을 통한 중복 방지

수집에 성공한 주문건에 한하여, 협력업체 시트의 N열(상태)에 **'접수완료'**라고 자동 기록하여 다음 수집 때 또 들어오지 않도록 잠금 장치를 겁니다.

주의사항: 고유 ID(UID)의 중요성
모든 수집과 매칭은 M열의 고유ID를 기준으로 처리됩니다. 이 고유 ID는 시트에 수식이 깨져서 `#REF`나 빈칸으로 보일 경우 매칭되지 않으므로, 항상 정상적으로 동작하는지 주의해서 보아야 합니다.
[자가 체크] '대리판매 발주수집' 메뉴가 실행되었을 때, 협력업체 시트에는 어떤 흔적이 남게 되나요?
제 2강

2강. 이카운트 판매현황 갱신

중앙 발주허브에 취합된 발주 데이터들 중, ERP에 등록되지 않은 데이터를 추출하여 업로드 엑셀용 시트를 갱신하는 단계입니다.

참조 시트
본사 시트: 협력업체_발주허브 탭
업로드 시트: 판매현황_갱신 탭
실행 조건
수집 완료 후, 이카운트에 매출/판매 기록을 대량 등록하고자 할 때
수행 후 결과
업로드 엑셀 파일 기입 완료
허브 P열 ➔ "이카운트 업 완료" 잠금
동작 특징과 이중등록 방지(Lock)

이 기능은 이카운트 ERP 연동의 안정성을 높이기 위해 **이중 업로드 방지 락** 구조를 가지고 있습니다.

1
미업로드 주문 감지

허브 데이터 중 P열(이카운트 업 완료 여부)이 완전히 **비어있는** 주문 행만 선별하여 추출합니다.

2
양식 재구성

추출된 데이터를 이카운트 판매업로드 서식 규격에 맞춰 각 열에 자동 기입합니다.

3
허브 락 처리

처리가 무사히 끝난 허브 상의 주문 행 P열에 즉시 **'이카운트 업 완료'** 플래그를 표기하여 락을 걸며, 이로 인해 다음번 갱신 시 중복 기입되지 않도록 격리됩니다.

실무 팁: 수동 재발송이 필요한 경우
특정 주문건을 수정하여 이카운트에 다시 올려야 한다면, 협력업체_발주허브의 해당 주문 줄 P열의 **'이카운트 업 완료' 글자만 지운 뒤** 갱신 기능을 재실행하면 됩니다.
[자가 체크] 2단계 판매현황 갱신을 실행할 때, 중복 업로드를 막아주는 장치는 무엇인가요?
제 3강

3강. 대리공급업체로 발주 Push

본사 '대리공급업체 발주' 시트에 들어온 발주 데이터를 분석하여, 해당 품목을 제조/공급하는 협력업체의 전용양식 탭으로 주입하는 과정입니다.

참조 시트
본사: 대리공급업체 발주 / 별칭 테이블
업체 시트: 전용양식 탭
실행 조건
수집 완료된 대리공급 발주 주문들을 협력 공급사에게 전송해야 할 때
수행 후 결과
공급업체 전용양식에 기입 완료
임시기록 ➔ "발주완료" 표시
상세 구동 순서

본사 메뉴의 3️⃣ 대리공급업체로 발주 Push 실행 시 아래 로직으로 분배됩니다.

1
품목코드 분석 및 분기

대리공급업체 발주 시트의 D열(이카운트코드)의 맨 앞 2자리 접두사(Prefix, 예: NK, HR, BW)를 추출하여 공급업체 파일을 선별합니다.

2
협력업체 전용 코드로 치환 (별칭 테이블)

대리발송 별칭 테이블 시트를 조회하여, 본사의 품목코드를 공급 협력사가 실제로 이해하고 사용하는 품목명 및 상품코드 규격으로 변환하여 기입합니다.

3
중복 발송 완전 필터링

각 행의 사방넷 번호(UID)와 치환된 품목코드를 결합하여 중복 식별용 복합 고유키(UID|품목코드)를 생성합니다. 해당 업체 시트의 49번째 열(AX열)을 검색하여 이 고유키가 존재하는 건은 누락하고 새 발주만 전달합니다.

4
서식 자동 정리 및 백업

연락처 숫자 0 누락을 방지하도록 텍스트서식 지정 후 주입하며, 전송 완료된 건은 본사의 대리공급_임시기록 시트에 진행상태를 **'발주완료'**로 백업합니다.

실행 조건 추가 설명: B6 거래처 코드
Push 과정 중 공급업체 시트의 거래처 코드가 깨지거나 서식이 일반 숫자로 되어 앞자리 "0"이 탈락되는 경우(예: 이카운트 고객코드 `01392` ➔ `1392`로 변형), 매핑에 에러가 발생할 수 있습니다. 스크립트가 실행될 때 자동으로 이 셀들을 텍스트 서식 `@`로 지정하여 유지합니다.
[자가 체크] 발주 Push 실행 시, 본사의 이카운트 품목코드가 협력사의 개별 코드로 바뀌는 원천 정보는 어디서 참조되나요?
제 4강

4강. 송장 수집 및 배포 처리

공급사에서 전용 시트에 입력한 배송 송장 정보를 가져오고, 대리판매처 시트에는 역으로 송장 정보를 뿌려주는 흐름입니다.

참조 시트
본사: 협력업체_발주허브 / 임시기록
업체 시트: 전용양식 / 발주 및 송장조회
실행 조건
공급사 송장 입력 완료 시 ➔ 수집
수집 완료 후 판매처 전달 ➔ 배포
수행 후 결과
허브 송장 기입 및 상태 "발송완료"
판매사 시트에 송장 자동 기록
송장 수집 및 배포의 투웨이(Two-way) 연동 구조

송장 데이터는 공급사에서 회수하여 판매사 시트로 밀어주는 다단계 파이프라인으로 구성되어 있습니다.

1
송장 번호 감지 및 회수

5️⃣ 허브로 송장 수집 실행 시, 공급 협력사 전용 파일들을 조회해 전용양식 K열에 송장이 적혀있는지 수집합니다. (AX열의 UID 키를 기반으로 매칭함)

2
본사 허브 상태 업데이트

회수한 송장을 중앙 협력업체_발주허브의 N열에 대입하고 O열 상태를 **'발송완료'**로 바꿉니다. 더불어 대리공급_임시기록의 진행상태도 **'송장수집'**으로 변경합니다.

3
판매처로 송장 배포

7️⃣ 대리판매업체로 송장 배포를 실행하면 본사 허브에 매칭된 송장을 각 판매처 시트의 `발주 및 송장조회` 탭 송장번호 칸(K열)에 주입합니다.

기타 수집 방식: 카카오 송장매칭 & Gmail 수집
• **카카오 송장매칭**: 협력사가 엑셀을 쓰지 않고 카톡 등으로 보낸 경우 사이드바에서 일련번호로 수동 매칭합니다.
• **Gmail 송장 수집**: 냅킨코리아와 같이 이메일로 송장을 주는 경우 메일함을 긁어와 자동 매칭 처리합니다.
[자가 체크] 공급사가 입력한 송장이 수집 완료되면, 중앙 허브의 '상태(O열)'는 어떻게 업데이트되나요?
제 5강

5강. 마감탭 정리와 아카이브

매일의 운영이 끝나거나 월말이 되었을 때, 시트 처리 속도를 높이고 데이터를 보존하기 위해 기처리 완료된 데이터를 백업하고 본래 탭을 비우는 과정입니다.

참조 시트
본사: 발주허브 / 대리공급 임시기록
업체 시트: 각종 발주/공급 탭
실행 조건
당일 영업 종료 후 또는 정기 마감 정산일 도래 시
수행 후 결과
원본 발주 데이터 ➔ 행 삭제 (비우기)
이전 데이터 ➔ 마감 탭으로 보관 이체
일괄 마감 이동의 4단계 연쇄 작동 방식

메뉴에서 🔄 [일괄] 대리판매+대리공급+발주허브 순차적 마감이동을 실행하면 백그라운드에서 다음 4가지 동작이 연속해서 수행됩니다.

순서 대상 범위 실행 필터 조건 마감 후 이체 경로

2단계 대리공급 마감 오늘 이전 날짜 & (**송장번호 기입** 또는 **상태 발송완료**) 건 업체 시트 내 (YYYY년 M월) 전용발주 마감 탭

6강. 시트 복구 도구 사용법

협력업체들이 개별 시트를 임의로 편집하다 수식이 깨지거나, 양식 디자인이 망가졌을 때 본사 관리자가 일괄 복구하는 긴급 AS 조치 도구입니다.

참조 시트
본사: AS / 진단 ➔ 복구 도구 메뉴
대상: 문제가 발생한 협력업체 파일
실행 조건
협력사 파일 내 #REF!, Spill! 오류 등 시각적/기능적 에러 민원 접수 시
수행 후 결과
대상 업체의 수식/헤더/스크립트 가동 상태 정상 복구
복구 도구 카테고리 분류 및 대처 방법

복구 도구 메뉴를 클릭하면 체크박스가 포함된 **일괄 복구용 팝업 창**이 뜹니다. 문제가 발생한 업체를 다중 선택한 뒤 적절한 카테고리를 선택해 실행합니다.

복구 종류 주요 복구 상세 대응 조치 상황
📊 수식·단가 복구 단가조회 탭의 `ARRAYFORMULA` 및 발주 탭의 Spill 연동 수식 재주입 사용자가 단가 3행의 수식을 수동으로 삭제하여 단가나 품명이 나오지 않는 상태일 때 실행
📋 헤더·양식 복구 발주 탭 1행 헤더 강제 기입 및 행 고정, 조건부 서식 재적용 협력사가 1행 헤더의 이름(예: 이카운트코드)을 바꾸거나 삭제하여 에러가 날 때 실행
📝 스크립트 재설치 이카운트코드 입력 시 작동하는 자동완성 작동 코드(onEdit) 재생성 사용자가 코드를 쳤는데도 품명이나 수량 등이 자동완성되지 않고 반응이 없을 때 실행
🗃️ 데이터 보정 거래처 코드 포맷 텍스트 전환 및 L열 단가 오류 강제 갱신 거래처코드 맨 앞 0이 유실되었거나, 기타 데이터 깨짐이 의심될 때 실행
성능 팁: Stuck 현상 발생 시
여러명이서 동시 기입할 때 동기화가 풀리지 않고 먹통이 되는 Stuck 현상이 생기면, 메뉴의 `🔓 동기화 락 강제 해제`를 가동하여 락 걸린 캐시 키를 날려주면 바로 풀립니다.
