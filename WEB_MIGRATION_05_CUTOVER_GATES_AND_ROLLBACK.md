# Pack2U 전환 게이트 및 롤백 기준 (Go/No-Go)

## 1. 목적
- 웹 단일화 전환 시점을 수치 기반으로 판단하고, 실패 시 즉시 복귀 절차를 표준화한다.

## 2. 단계별 게이트

## Gate A (Phase1 -> Phase2)
- 조건:
  - 데이터 모델/키 정책 문서 승인
  - 브리지 PoC에서 샘플 3업체 동기화 성공
  - CUST_CD 누락 0건
- No-Go 시 조치:
  - 매핑 정비 후 재검증

## Gate B (Phase2 -> Phase3)
- 조건:
  - 읽기 대시보드와 시트 집계 diff ±0.5% 이내(10영업일)
  - 브리지 오류율 1% 미만
  - 운영팀 “조회 품질 OK” 승인
- No-Go 시 조치:
  - 웹 조회만 유지, 입력 이관 연기

## Gate C (Phase3 -> Phase4)
- 조건:
  - 웹 입력 기능 사용 비율 70% 이상
  - 충돌건 중 미해결 0건
  - 송장/상태 전환 SLA 충족(30분 내 95%)
- No-Go 시 조치:
  - 웹 입력 범위를 제한하고 시트 우선 유지

## Gate D (Phase4 -> Phase5)
- 조건:
  - 이카운트 업로드 성공률 98% 이상(2주)
  - 업로드 멱등 충돌 0건
  - 월정산 결과 diff ±0.3% 이내
- No-Go 시 조치:
  - 업로드만 시트 fallback 허용

## Gate E (최종 Cutover)
- 조건:
  - 3주 연속 핵심 KPI 충족
  - S1 장애 0건, S2 장애 주당 2건 이하
  - 운영/관리자 승인 완료

## 3. 롤백 트리거
- 즉시 롤백(S1):
  - 업로드 전면 실패
  - 주문 누락/중복 대량 발생
  - CUST_CD 매칭 오류로 회계 영향 발생
- 조건부 롤백(S2):
  - 특정 업체군 반복 장애
  - 브리지 충돌 누적(당일 10건 이상)

## 4. 롤백 레벨
- **L1 (기능 롤백)**: 웹 쓰기 중지, 읽기 유지
- **L2 (도메인 롤백)**: 업로드/정산만 시트 복귀
- **L3 (전체 롤백)**: 시트 단독 운영 복귀, 웹은 모니터링 전용

## 5. 롤백 실행 절차 (30분 내)
1. Incident 선언(S1/S2 구분)
2. 브리지 쓰기 비활성화(`bridge.write_enabled=false`)
3. 운영 공지(영향 범위/복구 ETA)
4. 시트 fallback 메뉴 실행(업로드/발주 경로 복귀)
5. 데이터 무결성 점검(diff 리포트)
6. 재발 방지 항목 등록

## 6. 복구(Forward fix) 조건
- 원인 패치 완료
- 동일 시나리오 재현 테스트 통과
- 파일럿 업체 1일 재검증 성공
- 승인 후 단계적 재개(업체군 20% -> 50% -> 100%)

## 7. 의사결정 매트릭스
- 승인권자:
  - 운영 리드: L1 결정
  - 시스템 오너: L2 결정
  - 대표/총괄: L3 결정
- 기록:
  - 사고번호, 영향건수, 손실추정, 조치, 재발방지

## 8. 체크리스트 (Cutover Day)
- [ ] 브리지/큐 대기 0건 확인
- [ ] 매핑 시트 CUST_CD 누락 0건 확인
- [ ] 자동예약 트리거 상태 확인
- [ ] 업로드 테스트 1건 성공
- [ ] 롤백 스위치 위치 운영팀 공유
- [ ] 종료 후 24시간 집중 모니터링 계획 배포
