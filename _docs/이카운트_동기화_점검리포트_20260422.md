- **Zone**: `POST https://oapi.ecount.com/OAPI/V2/Zone` — body `{COM_CODE}` → `Data.ZONE`

- **업무 호출**: `https://oapi{ZONE}.ecount.com/OAPI/V2/...?SESSION_ID=...` + JSON body

### 🟡 High - 신뢰성 (업무 사고 직결)

#### R1. `Status === "200"`만 보고 FailCnt/부분실패 무시 ⭐ 최우선
- **위치**: `sendSales.gs:133-151`, `sendPurchase.gs:130-149`, `sendInventory.gs:136-155`, `salesUploadFromIntegratedOrders.gs:332-365`
- **시나리오**:
  1. 10건 묶어 전송 → HTTP 200 + `Data.FailCnt: 3` 응답
  2. 현재: ✅성공으로 기록 → `buildUploadedKeyMap_`에 SUCCESS
  3. 다음 실행: "이미 업로드됨"으로 제외 → **실패 3건은 영원히 이카운트에 반영 안 됨**
- **조치**: 공통 `evaluateEcountResponse_(result)` 판정 함수. 규칙: HTTP 200 AND `Status === "200"` AND `Data.FailCnt === 0` AND `!result.Error`. `ResultDetails`에서 라인별 메시지 추출해 시트에 기록
- **정책 결정 필요**: 통합판매 그룹 10건 중 3건 실패 시
  - (a) 그룹 전체 FAIL 기록 → 다음 실행 10건 재시도 (안전, 3건 중복 가능)
  - (b) 성공 7건만 SUCCESS, 실패 3건만 재시도 (라인별 식별자 필요)
  - **권장**: (a) 먼저, (b)는 응답 구조 파악 후 점진 이관

#### R2. `LockService` 없음 → 이중 전송 가능
- **범위**: 전 전송/업로드 함수
- **시나리오**: 스케줄(10시/14시) + 수동 실행 겹침 → 동일 `orderLineKey`가 일시적으로 둘 다 "미전송"으로 보여 **이카운트에 중복 전표**
- **조치**: 진입부 `LockService.getScriptLock().tryLock(10000)` + `try/finally`로 해제 보장

#### R3. 전송 경로에 `fetchWithRetry` 미적용
- **위치**: `sendSales/Purchase/Inventory/Item.gs`, `pushToEcount.gs`, `salesUploadFromIntegratedOrders.gs`
- **위험**: 일시 오류(429/5xx/timeout)가 영구 실패로 기록
- **조치**: 기존 `ecount.gs`의 `fetchWithRetry` 재사용 또는 공통 래퍼

#### R4. 배치 중간 세션 만료 시 재로그인 없음

## 4. 사전 결정이 필요한 정책

### 4-1. R1 정책 — 부분 실패 시 처리 방식
- **(a)** 그룹 전체 FAIL 기록 → 다음 실행에서 재시도. 안전하지만 **이미 성공한 건은 이카운트에 중복 가능**
- **(b)** 성공 라인만 SUCCESS 기록, 실패 라인만 재시도. 이카운트 응답의 라인별 식별자 구조 파악 필요
- **권장**: **(a) 먼저 도입** → 응답 구조 분석 후 (b) 점진 이관

### 4-2. S1 이관 시점
- ScriptProperties 값을 **먼저 저장** → 코드 배포 → 검증 → 하드코딩 제거
- 순서 어기면 전 이카운트 기능 중단 위험

### 4-3. R9 중복방지 강화 범위
- 주문라인키를 `PropertiesService`에 월 단위로 누적 기록
- 월 롤오버 시 이전 달 데이터는 정산 스냅샷에만 보관
