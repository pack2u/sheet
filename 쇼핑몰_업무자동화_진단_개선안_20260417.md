# 쇼핑몰 업무 자동화 진단 보고서

작성일: 2026-04-17  
대상: `Pack2U_상품정보sheet` (Google Apps Script 기반 운영 자동화)

## 1) 현재 자동화 구조 요약

이 폴더의 자동화는 크게 4개 축으로 구성되어 있습니다.

- 이카운트 동기화: `ecount.gs`, `moveToEcount.gs`, `fetchItem.gs` 계열
- 단가 허브/배포 자동화: `priceManager.gs`
- 발주/송장 통합 허브: `orderSyncManager.gs`
- OCR 송장 매핑: `송장시스템/autoTracking.gs`

핵심 흐름은 아래와 같습니다.

1. 이카운트 API에서 품목/재고 로딩  
2. `상품정보` 및 허브 시트(`전체 그룹 단가표`) 갱신  
3. 공급처/판매처 독립 배포 시트로 단가/주문 양식 배포  
4. 배포 시트 주문을 `통합 발주 DB`로 취합  
5. 공급사 전송 -> 송장 취합 -> 배포 시트 역푸시  
6. 월별 아카이빙 및 대시보드 집계

운영 관점에서 이미 상당히 높은 자동화 수준이며, 특히 헤더 동적 탐지/중복 방지/트리거 분리는 실무 친화적으로 잘 구현되어 있습니다.

## 2) 주요 문제점 (우선순위 순)

### P0. 보안 취약점 (즉시 조치 필요)

- `ecount.gs`, `priceManager.gs`에 `API_CERT_KEY`, `COM_CODE`, `USER_ID`가 평문 하드코딩
- 외부 스프레드시트 ID가 다수 코드에 직접 노출
- 스크립트 저장소 접근 권한이 넓으면 계정 탈취/오남용 위험이 매우 큼

영향:

- 키 유출 시 ERP 데이터 조회/변경이 제3자에 의해 가능
- 계정 잠금/데이터 오염/감사 추적 불가 리스크 발생

### P1. 전역 함수 충돌 위험 (GAS 특성상 치명적)

- `onOpen()`이 `menu.gs`와 `송장시스템/autoTracking.gs`에 중복 정의
- `verifyZoneAPI()`, `login()`이 `ecount.gs`와 `priceManager.gs`에 중복 정의

영향:

- 마지막 로드 파일 기준으로 함수가 덮여 실행될 수 있음
- 메뉴 누락, 로그인 동작 불일치, 디버깅 난이도 상승

### P1. 고정 열 인덱스/시트 구조 의존도가 높음

- 문서/코드 모두 열 위치 변경 금지 전제를 강하게 사용
- 일부 로직은 동적 매핑을 쓰지만, 여전히 다수 구간이 절대 위치 인덱스 기반

영향:

- 운영자가 열 1개만 수정해도 연쇄 장애 가능
- 신규 담당자 온보딩/인수인계 난이도 상승

### P2. 단일 스크립트에 역할 과밀

- `priceManager.gs`, `orderSyncManager.gs`에 생성/동기화/UI/보호/복구/마이그레이션이 혼재

영향:

- 변경 범위 예측 어려움
- 테스트 포인트 분리가 어렵고 회귀 가능성 증가

### P2. 운영 로그/감사 체계 부족

- `ui.alert` 중심, 예외/처리량/실패 건수를 구조화 저장하는 표준 로그가 약함
- 일부는 `console.error`만 존재하여 비개발자 관제 한계

영향:

- 장애 원인 추적 시간 증가
- 배치 실패 탐지 지연

## 3) 개선 방안

## 3-1. 단기(1~2주) - 안정화

- 비밀정보 분리
  - API 키/코드/시트 ID를 `PropertiesService`로 이관
  - 운영/테스트 값을 분리(`ENV=prod/stage`)
- 함수 네이밍 충돌 제거
  - `onOpenMain`, `onOpenTracking` 등으로 명시적 분리
  - 공통 엔트리에서 메뉴를 합쳐 호출
- 공통 API 모듈화
  - `verifyZoneAPI`, `login`, 공통 `fetchJson` 유틸 단일화
  - `Expect: ""` 헤더 정책 공통 래핑
- 표준 로그 시트 도입
  - `시스템_로그` 탭 생성
  - 실행ID, 함수명, 시작/종료시각, 처리건수, 실패사유 기록

## 3-2. 중기(3~6주) - 구조 개선

- 업무별 모듈 분리
  - `core/` (공통 유틸), `ecount/`, `pricing/`, `order/`, `tracking/`, `ops/`
- 데이터 계약(Contract) 정의
  - 필수 헤더/데이터 타입/상태코드 맵을 JSON 상수로 중앙화
- 실패 격리
  - 공급사별 처리에서 try/catch를 독립 처리하고 결과 리포트 누적
- 배치 재시도 정책
  - 실패 건만 재처리 가능한 큐 시트(`재처리큐`) 도입

## 3-3. 장기(6주+) - 확장성

- Apps Script + 별도 백엔드(Cloud Run/Functions) 하이브리드 검토
  - 대량 처리/OCR/알림/로그 집계를 서버로 분리
- 이벤트 기반 관제
  - 심각도별 알림(Slack/Telegram) 및 일일 리포트 자동 발송
- 권한 모델 정교화
  - 운영자/실무자/공급사별 편집 범위 템플릿화

## 4) 추천 신기능

### 4-1. 운영자 관제 대시보드 2.0

- 오늘 처리량, 미매칭 송장, 재고부족, API 실패율, 업체별 SLA를 단일 화면 제공
- "어디가 막혔는지"를 업무 단계별로 표시

### 4-2. 예외건 자동 큐 + 원클릭 재처리

- 매칭 실패/전송 실패 건을 자동 큐잉
- 운영자가 행 선택 후 "재처리" 버튼만 누르면 재시도

### 4-3. 송장 매칭 신뢰도 점수화

- OCR 매칭 결과에 점수(예: 0~100) 부여
- 저신뢰(예: 60 미만)만 별도 검수 탭으로 분리

### 4-4. 주문 상태 머신 표준화

- 상태값을 자유 텍스트가 아닌 Enum 기반으로 통일  
  예: `RECEIVED`, `WAIT_STOCK`, `SENT_SUPPLIER`, `INVOICE_MAPPED`, `SHIPPED`, `CANCELLED`

### 4-5. 변경 이력(Audit Trail) 자동 기록

- 핵심 열(상태, 송장, 단가, 재고) 변경 시
  - 변경 전/후
  - 변경자
  - 변경시각
  - 변경출처(수동/배치)를 기록

## 5) 실행 로드맵 (권장)

5~6주차
